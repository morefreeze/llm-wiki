# 深度拆解 Hermes Agent 的记忆系统：它如何修正 OpenClaw 的误区

作者：Manthan Gupta（宝玉翻译整理）
来源：https://baoyu.io/blog/2026-04-29/manthanguptaa-2034849672985288957
原文：https://x.com/manthanguptaa/status/2034849672985288957
日期：2026-04-29

---

Hermes Agent 是开源的，代码库和文档公开，可以直接翻看代码路径——从如何构建提示词状态、持久化会话，到如何清理记忆和查询历史对话。

**Hermes 拥有的不是一套记忆系统，而是四套：**

1. 存储在 `MEMORY.md` 和 `USER.md` 中、经过高度浓缩的**提示词记忆**。
2. 通过 `session_search` 调用的 **SQLite 历史会话存档**（可搜索）。
3. 像**程序记忆**（Procedural Memory）一样运作的**智能体技能管理**。
4. 可选的 **Honcho** 层，用于更深层的**用户建模**（User Modeling）。

把这些设计联系在一起的核心逻辑非常简单：**保持提示词稳定以便利用缓存（Caching），其他一切繁杂信息都交给工具。**

---

## Hermes 的上下文结构

系统提示词（System Prompt）的组装顺序：

```
[0] 默认智能体身份
[1] 工具使用行为指南
[2] Honcho 集成模块（可选）
[3] 可选系统消息
[4] 固化的 MEMORY.md 快照
[5] 固化的 USER.md 快照
[6] 技能索引
[7] 上下文文件（AGENTS.md, SOUL.md 等规则文件）
[8] 日期/时间 + 平台信息
[9] 对话历史
[10] 当前用户消息
```

Hermes 正在针对大模型供应商的**提示词缓存（Prompt Caching）**机制进行优化，目标：**让稳定的前缀部分尽可能长时间地保持不变。**

这一个决定就解释了 Hermes 大部分的记忆架构——如果某条信息每一轮对话都要用到，尽量缩小并注入提示词；如果信息量大、属于历史旧账或偶尔才有用，踢出提示词，改用"按需检索"。

---

## 第一层：固化的提示词记忆（Hot Memory）

Hermes 将持久记忆存储在 `~/.hermes/memories/` 下的两个文件中：

| 文件 | 用途 | 限制 |
|------|------|------|
| `MEMORY.md` | 智能体笔记：环境、规范、工具怪癖、教训 | 2,200 字符 |
| `USER.md` | 用户画像：偏好、沟通风格、身份信息 | 1,375 字符 |

合计约 1,300 个 Token——刻意保持极小。

**会话开始时加载两个文件，固化快照，整个会话期间不变。** 会话中途写入的记忆立即存入硬盘，但不改变已生成的系统提示词——只有在开启新会话，或触发"压缩（Compression）"导致的提示词重建时才生效。

### 设计亮点

1. **使用字符限制而非 Token 限制**：让记忆逻辑与模型无关，不需要调用特定模型工具就能判断是否存满。
2. **简单的分隔符文件格式**：条目之间用 `§` 分隔，纯文本，无向量数据库，无二进制存储。
3. **记忆是"精选状态"，不是"日记"** — 这是与 OpenClaw 最大的区别。

### 存什么/不存什么

**存储：**
- 用户偏好
- 环境事实
- 反复出现的错误修正
- 稳定的规范

**不存储：**
- 任务进度
- 会话结果
- 临时的待办事项（TODO）

### `memory` 工具

三种操作：`add`（添加）、`replace`（替换）、`remove`（移除）。
- `replace` 和 `remove` 使用**子字符串匹配**，不需要记住内部 ID。
- 系统会拒绝完全重复的内容，并拦截危险信息（防止**提示词注入**、凭证泄露或隐藏 Unicode 字符）。

## 第二层：`session_search` — 情景回溯（Cold Storage）

所有过去的会话存储在 SQLite 数据库中，拥有完整的索引和搜索功能。

工作流程：
1. 在过去的消息中进行全文搜索
2. 按会话分组结果
3. 加载匹配度最高的会话
4. **使用一个便宜的辅助模型对这些会话进行摘要总结**
5. 将精炼后的回顾内容返回给主模型

比盲目地把长篇累牍的历史塞进每一个提示词更便宜且高效。

---

## 第三层：压缩与记忆冲刷（Memory Flush）

当会话变得太长，Hermes 会压缩对话中间的部分。但摘要是有损的，重要事实可能丢失。

**于是，Hermes 会先进行一次"记忆冲刷"：**

在压缩之前，发送指令给模型：

> "会话即将压缩，请保存任何值得记住的东西。优先保存用户偏好、修正建议和重复模式，而非具体的任务细节。"

然后运行一次额外的模型调用，只开启 `memory` 工具。模型觉得有什么该留下来，就在对话被"洗掉"之前写入 `MEMORY.md`。

---

## 第四层：Skills — 程序记忆（Procedural Memory）

Hermes 不仅能记住事实，还能记住**技能**。

技能存储在 `~/.hermes/skills/` 下。当 Hermes 发现了一个复杂的流程、修复了一个棘手的问题或学会了更好的方法时，可以将其保存为"技能"。

大多数记忆系统只关注"语义回溯"（名字、偏好、事实），但智能体还需要记住**如何做事**。

为了效率，Hermes 不会把所有技能都塞进提示词，而是只放一个**技能索引**，只有在需要时才加载具体的技能内容。

---

## 可选第五层：Honcho — 深层用户建模

Honcho 是可选层：跨设备、跨平台的记忆连续性，构建复杂用户模型。

集成方式（不破坏提示词缓存）：
- **第一轮**：Honcho 的上下文织入系统提示词。
- **之后的对话**：Honcho 的回溯内容附加在**当前用户的提问**后面，而不修改系统提示词。

**这确保了缓存依然有效，同时 AI 依然能读到最新的背景信息。**

---

## Hermes vs OpenClaw 的区别

| 维度 | OpenClaw | Hermes |
|------|---------|--------|
| 记忆主要存储 | 以 Markdown 为中心，日志和长效文件 | 严格限制的提示词记忆 + SQLite |
| 历史查询 | 可能每次都注入 | 只在需要时通过 session_search 搜索 |
| 缓存优化 | 较弱 | 核心设计目标 |
| 技能（程序记忆） | 无对应设计 | Skills 系统 |

**Hermes 更加关注缓存效率：** 不是所有东西都配住在"系统提示词"这个黄金地段。

---

## 总结：Hermes 做对了什么？

1. **冷热分离**：小规模提示词记忆（热，常驻）+ SQLite 搜索（冷，按需）
2. **缓存优先**：频繁改动提示词 = 延迟增加 + 成本上升；Hermes 尽量让提示词前缀稳定
3. **记忆的多样性**：个人画像（semantic）+ 情景回溯（episodic）+ 操作技能（procedural）+ 深层建模（user model）

> 真正的诀窍不是记住更多，而是在正确的层级、以正确的成本，记住正确的事情。
