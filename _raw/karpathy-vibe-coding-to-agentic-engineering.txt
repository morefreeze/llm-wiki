# Karpathy 最新访谈：Vibe Coding 只是开始，真正重要的是 Agentic Engineering

作者：宝玉（翻译/整理）
来源：https://baoyu.io/blog/andrej-karpathy-from-vibe-coding-to-agentic-engineering
日期：2026-04-29
原始视频：https://www.youtube.com/watch?v=96jN2OCOfLs
场合：Sequoia Capital AI Ascent，Karpathy 接受合伙人 Stephanie Zhan 访谈

---

Andrej Karpathy 说，他已经记不清上次修改 AI 生成的代码是什么时候了。

Karpathy 参与创建了 OpenAI，在 Tesla 领导过 Autopilot 视觉团队，去年一条推文发明了"凭感觉编程"（Vibe Coding）这个词，后来被 Collins 词典选为 2025 年度词汇。

## 要点速览

- **2025 年 12 月**是 Karpathy 个人的转折点：AI 输出从"有帮助但常要修补"变成"直接可用"，他进入完全凭感觉编程的状态。
- **Software 3.0 的关键**不是"用自然语言写代码"，而是通过 prompt 和 context 操作 LLM 这个新的信息处理解释器。
- **MenuGen 案例**让 Karpathy 意识到，一些 AI 应用不是会被做得更快，而是会被模型原生能力直接吞掉。
- **LLM 的能力高度不均匀**（锯齿状智能）：它可以重构 10 万行代码、找零日漏洞，却可能在"去 50 米外洗车该走路还是开车"这种常识题上犯错。
- **Vibe Coding 抬高所有人做软件的下限**；Agentic Engineering 则是在使用 Agent 提速时，保住专业软件的质量、安全和责任门槛。
- **人类不必再记住每个 API 细节**，但必须理解系统结构、底层机制和质量标准，否则无法监督 Agent。
- Karpathy 用"幽灵"形容 LLM：不是动物式智能，而是由人类文档、预训练统计和强化学习奖励塑造出的锯齿状实体。
- **智能变便宜后**，教育的重点不是抵制外包思考，而是确保理解仍然进入人的大脑。

---

## 【1】2025 年 12 月：一个程序员的投降

2025 年 12 月是转折点。最新模型生成的代码块开始"直接能用"。一开始让模型写一点，结果不错，越来越信任，最后进入 Vibe Coding 状态。

> 我记不得上一次我需要纠正它是什么时候了。然后我就越来越信任这个系统。

Vibe Coding：一种"凭感觉让 AI 写代码"的开发方式——人用自然语言持续提出意图，模型生成、修改、调试代码，人不再逐行写、逐行读 diff。

但 Karpathy 的重点已经不只是 Vibe Coding，而是 **Agentic coherent workflow**——一种更连贯的智能体工作流：模型不只是回答问题，而是能连续规划、写代码、调试、执行、根据环境反馈继续修正。

> 很多人去年体验到的 AI，还是一个类似 ChatGPT 的东西。但你真的必须重新看一眼，而且要看 12 月之后的版本。

过去程序员的速度取决于写多少代码、记多少 API。**现在，速度越来越取决于能否正确地指挥一组强大但会犯错的 Agent。**

---

## 【2】Software 3.0：给 Agent 复制粘贴一段文字，这就是编程

软件分期：
- **Software 1.0**：传统软件，人写显式代码，计算机按规则执行
- **Software 2.0**：神经网络时代，设计数据集和架构，通过训练得到模型权重（Karpathy 2017 年提出）
- **Software 3.0**：LLM 时代，LLM 变成可编程的计算机；通过 prompt、context window、文件、工具调用组织"上下文程序"

context window = 模型一次调用能看到的全部信息（指令、历史对话、文件、错误日志、代码片段、图片、工具返回结果）

> 现在的问题变成：哪一段文字应该复制给你的 Agent？这就是新的编程范式。

程序边界扩大：过去的程序是代码文件；现在，程序可能是一段说明、一个上下文窗口、一组工具权限、一个测试环境，外加模型内部已经学到的大量统计结构。

---

## 【3】MenuGen：这个 App 不应该存在

MenuGen：拍菜单照片，OCR 识别菜名，为每道菜生成图片，重新渲染菜单。Karpathy 用 Vibe Coding 做了出来。

但 Software 3.0 版本：直接把菜单照片交给 Gemini，Nano Banana（Google 图像生成能力）直接把菜品图叠加回菜单——原菜单仍在，菜品位置直接渲染进图片，中间所有步骤都消失了。

> 我的整个 MenuGen 都是多余的。它还停留在旧范式里。那个 App 不应该存在。

**关键商业判断：** 很多 AI 应用以为自己在做"更快的软件"（10步→3步），但 Software 3.0 里，模型输入输出可能直接覆盖整个任务，中间 App 的结构就失去必要性。

> 更令人兴奋的不是把已有东西做得更快，而是那些以前根本不可能存在的东西。

LLM Knowledge Bases 正是此类：把文章、文档、事实重新编译成个人或组织 wiki——传统程序天然不擅长，因为它需要理解文本之间的关系、重排序信息、生成新的知识结构。

---

## 【4】神经计算机：CPU 变成协处理器

设想：未来可能出现"神经计算机"——神经网络成为 host process，CPU、传统代码和工具调用变成协处理器，负责确定性任务。

设备接收原始视频/音频，神经网络理解当前场景，扩散模型为这一刻生成独特的 UI。用户看到的界面由模型实时生成，而非固定组件拼出来的。

Karpathy 自己加了限制：这种外推很怪，具体路径 TBD，不会一夜之间发生。更像是他用来解释方向感的心智模型。

---

## 【5】锯齿状智能：LLM 能力分布的高峰和断崖

**核心判断：**
> 传统计算机容易自动化你能写进代码的东西；这一代 LLM 容易自动化你能验证的东西。

- 传统软件自动化：规则清楚才能写代码（税率计算、排序、数据库查询）
- LLM 自动化：不需要把规则全部写出来，但需要某种方式判断输出好坏

数学题可以验证答案，代码可以跑测试，安全问题可以通过漏洞复现判断——这样的任务能进入强化学习（RL）环境：模型尝试解题→系统给奖励/惩罚→大量样本中优化行为。

这解释了"锯齿状智能"（jagged intelligence）：能力曲线不是平滑上升，而是有高峰和断崖。

**洗车题：** 我要去 50 米外洗车店洗车，应该开车还是走路？最先进的模型可能说走路（因为很近）——忽略了关键：你要洗的是车，车必须到洗车店。

> 一个最先进的模型可以重构 10 万行代码、找到零日漏洞，却告诉我应该走路去洗 50 米外的车。

如果任务落在模型训练和 RL 覆盖的能力回路里→像专家；落在数据分布外→可能出错，即使人类觉得很简单。

---

## 【6】能力来自数据决策，不是自然进化

GPT-3.5→GPT-4 国际象棋能力大提升，原因：有人在 OpenAI 决定把大量国际象棋数据加进了预训练。

> 某种程度上，我们完全受制于实验室给模型喂了什么数据。

实操含义：你的应用场景在覆盖的能力回路里→开箱即用；在外面→需要自己做微调，不要指望 LLM 一上来就会。

---

## 【7】创业机会：找一个还没被 RL 覆盖的可验证领域

创业方法：如果能构造大量、多样的强化学习环境（让模型尝试、失败、获得奖励），即便大实验室没把这个领域作为重点，也可能通过自己的微调和训练获得优势。

Karpathy 暗示有"被低估的 RL 机会"领域却不愿公开（停住了）。他的公司 Eureka Labs 做的是 AI 教育——教育恰好是一个输出可以被评估、可以被标准化考核的领域。

几乎所有事情最终都可能在某种程度上变得可验证：写作、设计这类主观任务，也可以想象用 LLM judges 形成近似评价。

---

## 【8】Vibe Coding vs Agentic Engineering：下限 vs 上限

**Vibe Coding 抬高的是下限：** 更多人可以用自然语言和 AI 做出软件。软件创造的入口变宽了。

**Agentic Engineering 保住的是上限：** 专业软件的质量、安全、责任门槛。不能因为用了 AI 就引入安全漏洞，不能因为模型写得快就降低质量门槛，不能因为代码是 Agent 生成的就没人负责。

> Vibe Coding 抬高的是所有人能做软件的下限；Agentic Engineering 要保住的是专业软件过去已有的质量门槛。

Agentic Engineering（智能体工程）：不是具体工具，而是工程纪律——如何设计、协调、监督一组 AI Agent，在不牺牲质量、安全、可维护性的情况下加速开发。

Agent 是"spiky entities"（有尖刺的实体）：能力很强，但会犯错，有随机性，不稳定。工程师的工作是把它们放进合适的流程：生成方案→写代码→跑测试→互相检查→有边界、有验证、有回滚。

> 10x 不是你获得的加速倍数。（真正熟练的人加速幅度远不止 10 倍）

---

## 【9】AI-native 工程师：面试要用大项目，不用算法题

AI-native 工程师：充分利用可用工具，投资工作流设置（就像过去工程师配置 Vim/VS Code）。

面试重构：算法题测不出在 Agentic Engineering 里高效工作的能力。更好的测试：让候选人做一个 Twitter clone，不仅要能跑，还要做得好、安全。然后用多个 Agent 作为"红队"去攻击这个网站。

评估核心：把模糊目标变成清晰规格；指挥 Agent 完成大规模实现；识别安全和架构风险；设置测试与验证；在模型生成的大量代码里保持质量判断；让最终系统经得起外部攻击和压力。

---

## 【10】人要负责规格（Spec），不只是细节

Agent 做得越多，什么人类技能更有价值？**品味、判断、审美、监督，以及规格设计。**

MenuGen 的实际 bug：用户用 Google 账号登录，用 Stripe 账号购买 credits。Agent 试图用 Stripe 邮箱去匹配 Google 邮箱来关联资金。正确做法是使用系统内部稳定的 persistent user ID。

> 你为什么要用邮箱地址来交叉关联资金？它们可以是任意的，你可以用不同的邮箱。这种做法太奇怪了。

代码能跑，但系统设计是错的。Agent 没有真正理解身份、支付和资金归属的风险。

**"细节可以外包，理解不能外包。"** API 名称可以忘（Karpathy 说他不再记 PyTorch/NumPy/pandas 之间的细碎 API 差异），但概念结构不能丢（tensor 是什么，view 和 storage 的关系，什么时候复制内存）。

---

## 【11】品味差距：能跑的代码和好代码不是一回事

模型生成的代码有时让 Karpathy "心脏病发作"：能跑，但臃肿，有很多复制粘贴，有别扭的抽象，结构脆弱，维护很难。

MicroGPT 项目：他想把 LLM training 简化到极致，让模型"再简化一点"，但模型做不到。"就像在拔牙。"这个任务走出了模型被 RL 覆盖的能力回路——模型擅长生成常见工程形态，不擅长极简、克制、优雅的抽象压缩。

---

## 【12】幽灵心智模型：LLM 不是动物

LLM 的两种心智模型对比：

**动物：** 进化、身体、环境互动、内在动机、好奇心、持续学习；会在世界中行动，被后果塑造。

**幽灵：** 大规模预训练（海量人类文档的统计结构）+ 强化学习 + 偏好数据 + 工具调用；不是动物式智能，而是由人类文档、统计模式和奖励函数塑造出的模拟实体。

实用原则：不要把 LLM 当动物。对它大喊大叫，不会让它因为害怕而更努力。鼓励它，也不是在激发内在动机。模型的行为来自统计模拟、上下文、工具、训练数据和奖励机制。

> 如果你对它大吼，它不会因此工作得更好或更差，也没有任何影响。

不要笼统地问"AI 聪不聪明"，要问它在哪些训练分布里强，哪些奖励信号塑造了它，在哪些任务上可能出现锯齿状断崖。

---

## 【13】Agent-first 基础设施：一切要重写

几乎一切都要重写。今天的工具、文档、服务和设置流程，仍然主要是为人设计的。

Karpathy 的期望：框架文档不应该告诉我该怎么做，而应该直接给我"复制粘贴给 Agent 的东西"。

**Agent-first infrastructure 的目标：** 把世界拆成 Agent 能读懂的输入，以及 Agent 能安全调用的动作接口。不是让 Agent 模拟人去点网页，而是让 Agent 直接理解状态、调用动作、收到反馈。

这意味着文档、API、权限、日志、部署、配置、账单、回滚，都要变得更适合 Agent 使用。

未来设想：每个人、每个组织都可能有自己的 Agent representation。安排会议、处理细节、协调事项，可能变成"我的 Agent 和你的 Agent 去谈"。

---

## 【14】智能变便宜后，最贵的是理解

> 你可以外包你的思考，但不能外包你的理解。

Karpathy 仍然是系统的一部分。信息必须进入他的脑子里。他感觉自己正在变成瓶颈：要知道到底在建什么，为什么值得做，怎样指导自己的 Agent。

这是他对 LLM Knowledge Bases 感兴趣的原因：把文章、资料、事实重新投影成 wiki，不是让 AI 代替理解，而是用 AI 增强理解。

**学习重心的变化：** 更少时间花在机械记忆和低层执行上，更多时间花在系统理解、问题定义、质量判断、因果关系和领域直觉上。

---

## 宝玉评论：三个值得注意的张力

1. **Karpathy 一边说代码丑得让他心脏病发作，一边说他自己已经停止修改了。** "信任并没有解决品味的问题"——能跑的代码和好代码不是一回事。这种"用着丑的，但用着"的状态，可能比任何 hype 都更接近真实的 Vibe Coding 现状。

2. **他暗示有"被低估的 RL 机会"领域却不愿公开。** 他的公司 Eureka Labs 做的是 AI 教育——教育恰好是一个输出可以被评估的领域。

3. **"外包思考但不能外包理解"这句话的有效期是有问号的。** Karpathy 承认，品味和判断之所以暂时不可替代，原因只是"实验室还没做"。所谓人类不可替代的部分，不是因为人类独一无二，而是因为训练方法还没到位。

**未来 6-12 个月的三个信号：**
1. 前沿实验室在编程/数学之外，往哪些领域注入 RL 数据
2. Agent-first 基础设施（部署、auth、payments）是否有第一波收敛
3. 模型下一代更新是否包含审美和代码质量相关的 RL 目标
