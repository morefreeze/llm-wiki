# LLM Wiki Pattern — 用 LLM 增量构建持久知识库

- 原作者：Andrej Karpathy
- 来源：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
- 日期：2026 年

## 核心思想

传统 RAG 的问题：每次提问都要从原始文档重新检索和拼凑知识，没有积累。

LLM Wiki 的方式：LLM **增量构建并维护一个持久化 wiki**——结构化的、互联的 Markdown 文件集合。新源加入时，LLM 不仅索引它，还：
- 阅读并提取关键信息
- 整合到现有 wiki 中（更新实体页、修正主题摘要、标记矛盾）
- 知识编译一次，持续保持最新

> **Wiki 是一个持久的、复利的知识产物。** 交叉引用已经建立，矛盾已经标记，综合分析已反映所有已读资料。

## 三层架构

1. **Raw Sources（原始资料）**：不可变，只读
2. **The Wiki**：LLM 生成的结构化页面（实体、主题、摘要、综合分析）
3. **The Schema**（如 AGENTS.md / CLAUDE.md）：告诉 LLM 如何维护 wiki

## 关键操作

### Ingest（摄入）
- 一次处理一个源，逐步积累
- 一个源可能影响 10-15 个 wiki 页面
- 人类主导筛选和方向，LLM 做整理和交叉引用

### Query（查询）
- LLM 搜索 index → 阅读相关页 → 综合回答
- **好的回答可以存回 wiki**，成为新的知识页面

### Lint（检查）
- 定期检查：矛盾、过时信息、孤立页面、缺失引用
- LLM 擅长建议新的探索方向

## 导航工具

- **index.md**：内容目录（按类别组织），LLM 每次摄入后更新
- **log.md**：时间日志（append-only），用 `## [日期] 操作 | 标题` 格式便于 grep

## 适用场景

- 个人知识管理（目标、健康、心理学）
- 深度研究（论文、报告 → 综合wiki）
- 读书伴侣（章节 → 人物、主题、情节 → 粉丝wiki级别）
- 团队知识库（Slack、会议记录 → 维护wiki）
- 竞品分析、尽职调查、旅行规划、课程笔记

## 关键引述

> "人类放弃 wiki 是因为维护负担的增长速度快于价值。LLM 不会感到无聊，不会忘记更新交叉引用，可以在一次操作中触及 15 个文件。"

> "这个想法在精神上与 Vannevar Bush 的 Memex (1945) 相关——一个私人的、精心策划的知识存储，文档之间有关联路径。他没能解决的是谁来维护。LLM 处理了这个问题。"

## 对我的启发

用 Logseq 替代 Obsidian 作为浏览层：
- `R/` 前缀管理资源页（`R___xxx.md` 文件，`[[R/xxx]]` 引用）
- LLM 维护 wiki 目录下的 Markdown 文件
- Logseq 的图视图（Graph View）替代 Obsidian 的图谱查看连接
- 日志用 `log.md` 对应 Logseq 的 journal 模式
