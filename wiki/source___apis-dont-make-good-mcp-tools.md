# APIs Don't Make Good MCP Tools

## 一句话总结
> 把现有 REST API 机械地转换成 MCP 工具是错误做法——工具数量爆炸引发认知负荷、JSON 格式浪费 token、API 响应淹没上下文窗口，正确做法是专门为 Agent 的约束和能力重新设计工具。

## 关键要点
1. **工具数量上限**：VS Code 硬限 128 个工具，但许多模型在远未达到上限前就开始出错；API 的大量端点一旦直接映射为工具，立刻触及这一瓶颈
2. **上下文窗口宽度问题**：API 返回 100 条记录 × 50 个字段，所有数据都压入 Agent 上下文，即便只需要其中 2 个字段
3. **上下文窗口深度问题**：单条记录大小不可预测，一个含长文本字段的记录可能独自占用 10 万 token
4. **JSON 格式低效**：相同数据量下，CSV/TSV/YAML 比 JSON 节省约 50% token；JSON 的键名重复和嵌套结构在 LLM 消费场景中是纯粹的浪费
5. **缺乏 Agentic 能力**：好的 MCP 工具应内置字段投影（只返回需要的字段）、结果截断/摘要、格式转换，以及主动引导后续工具调用的上下文建议
6. **Agent 自写代码的替代路径**：Claude Code 等 Agent 已能独立编写并执行调用 API 的脚本，随着沙箱技术成熟，MCP 抽象层本身的必要性值得重新审视

## 详细笔记

### 核心论点：为什么机械转换行不通

许多开发者在接触 MCP 时的第一直觉是：把已有的 REST API 的每个端点包装成一个 MCP 工具。这种做法逻辑上说得通（API 已有文档、类型、权限控制），但在 Agent 实际使用时会暴露四类根本性问题。

### 问题一：工具数量与认知负荷

**硬性约束**：VS Code 对 MCP 工具数量有 128 个的硬限制。

**软性瓶颈**：许多主流模型在工具数量远未达到 128 之前就开始出现工具选择错误率上升的问题——大量工具描述本身会产生"认知噪音"。

**API 的天然问题**：一个功能完整的产品 API 往往有数十甚至数百个端点（增删改查、过滤、批量操作……）。如果每个端点对应一个 MCP 工具，工具数量立刻爆炸，而且每个工具描述都要占用宝贵的上下文空间。

**正确做法**：将多个相关 API 端点合并为一个意图驱动的灵活工具，让工具的粒度对齐用户意图而非 API 结构。

### 问题二：上下文窗口宽度爆炸

**场景**：调用一个"列出记录"的工具，API 返回了 100 条记录，每条记录有 50 个字段。

**问题本质**：Agent 只需要其中的"姓名"和"状态"两个字段，但 API 把所有 5,000 个字段值都塞进了 Agent 的上下文窗口。

**影响**：
- 大量无用数据占用 token 预算
- 后续对话轮次的可用空间被压缩
- 模型处理冗余信息时出错率上升

**解决方向**：MCP 工具应支持**字段投影**（field projection）——Agent 在调用时声明只需要哪些字段，工具只返回这些字段的数据。

### 问题三：上下文窗口深度爆炸

**场景**：同一个工具，不同记录的大小差异极大。

**作者数据**：一条含有大型文本字段的记录可能独自占用 **100,000 token**，而另一条只占 10 token。

**问题本质**：API 不承诺返回数据的大小上限，而 Agent 的上下文窗口是硬性有限资源。一次工具调用就可能耗尽整个对话的 token 预算。

**解决方向**：
- 自动截断（truncation）：返回超过阈值时截断并提示
- 自动摘要（summarization）：大型文本字段替换为摘要
- 分页感知（pagination-aware）：工具内建分页逻辑，按需加载

### 问题四：JSON 格式的 Token 低效

**作者论证**：以"姓名 + 年龄"数据集为例，对比两种格式：

**JSON 格式**（token 高消耗）：
```json
[
  {"name": "Alice", "age": 30},
  {"name": "Bob", "age": 25}
]
```

**CSV 格式**（token 节省约 50%）：
```
name,age
Alice,30
Bob,25
```

**原因分析**：
- JSON 的键名在每条记录中重复出现（`"name"`, `"age"` 在每行都有）
- JSON 的引号、花括号、方括号等结构符号本身占用 token
- CSV/TSV/YAML 避免了键名重复，结构更紧凑

**结论**：对于 LLM 消费场景，MCP 工具应默认将结果序列化为 CSV、TSV 或 YAML，而非 JSON。

### 问题五：缺失的 Agentic 能力

机械转换的 API 工具还欠缺一类关键能力：**主动引导 Agent 后续行动**。

好的 MCP 工具不只是返回数据，还应该：

- **返回结构化数据 + 后续工具调用建议**：例如，"你刚查询了用户列表，下一步可能需要调用 `get_user_details` 工具获取详情"
- **提供工具链条 hint**：帮助 Agent 理解哪些工具组合使用效果更好
- **内建错误恢复建议**：调用失败时提示 Agent 可以尝试哪种替代路径

这种"工具返回数据 + 元建议"的模式，才能充分发挥 Agent 的链式推理能力。

### 问题六：Agent 自写代码的替代路径

作者提出了一个更激进的视角：像 Claude Code 这样的现代 Agent 已经具备**自主编写并执行 API 调用脚本**的能力。

随着沙箱（sandboxing）技术的成熟，Agent 可以：
1. 直接读取 API 文档
2. 生成调用代码
3. 在沙箱中执行
4. 处理结果

这意味着：在某些场景下，精心设计的 MCP 工具层可能最终被"Agent 直接写代码调 API"所取代。MCP 工具在短期内的价值在于降低门槛，但其长期地位值得持续关注。

### 正确设计 MCP 工具的原则

根据文章的论证，作者隐含的设计原则可总结为：

| 原则 | 说明 |
|------|------|
| **按意图分组** | 合并相关端点，减少工具总数 |
| **字段投影** | 只返回 Agent 请求的字段 |
| **大小预算** | 内建截断/摘要，保证 token 可控 |
| **格式优化** | 优先 CSV/TSV/YAML，而非 JSON |
| **后续引导** | 在响应中附带后续工具调用建议 |
| **分页感知** | 工具自己处理分页，而非让 Agent 管理 |
