- # Coding Agents 101
- ## 一句话总结
  > Coding agent 不是全自主的——它需要人类战略性地引导、设置检查点、验证结果；资深工程师上手最快，因为他们懂得如何清晰指挥并辨别质量。
- ## 关键要点

  1. **指定"如何做"而非只说"做什么"** — 给 Agent 提供具体实现路径（测哪些功能、怎么 mock）而非模糊目标，减少低质量输出和返工
  2. **指明起点** — 引用相关文档、代码库位置、关键组件，最小化 Agent 浪费在定位上的精力
  3. **防御式提示（Defensive Prompting）** — 像给实习生 briefing 一样主动说明容易混淆的地方，包括技术前提（如重新编译步骤）
  4. **提供反馈回路** — 给 Agent 配备类型检查器、linter、单元测试、CI 权限；Agent 从错误信息中自我修正的能力是其"魔法"所在
  5. **检查点架构（Checkpoint Architecture）** — 将复杂任务结构化为"计划→实现分块→测试→修复→检查点审查→下一分块"，防止问题跨层级级联
  6. **目标约 80% 时间节省，而非完全自动化** — 最终质量保证仍需人类验证
  7. **知道何时放弃** — Agent 在"无限循环"或"忽视指令"时应立即终止，重新出发往往比继续迭代更快
- ## 详细笔记
- ### 核心定位

  Coding agent 是强力工具，但**并非完全自主**。它需要战略性的人类监督。资深工程师采用速度最快，因为他们理解如何通过清晰的方向和验证来有效引导 AI。文章的核心论点是：**如何与 Agent 协作** 比 Agent 本身的能力更决定结果。
- ### 基础提示原则（Four Essential Prompting Rules）

  #### 1. 指定"如何"而非只说"什么"
  - 错误示范："添加单元测试"
  - 正确做法：说明需要测试哪些功能、识别边界情况、明确 mock 需求
  - 目的：防止浪费代码审查周期，减少低质量输出

  #### 2. 指明起点
  - 引用相关文档、代码库路径、关键组件
  - 减少 Agent 在信息定位上的无效消耗
  - "最小化浪费的精力和混淆"

  #### 3. 防御式提示（Defensive Prompting）
  - 像给实习生 briefing 一样预判混淆点
  - 主动澄清技术前提条件，例如："修改后需要重新编译"
  - 先发制人地处理歧义，而不是等 Agent 出错后再纠正

  #### 4. 提供反馈回路
  - 给 Agent 配备：类型检查器、linter、单元测试、CI 访问权限
  - 关键洞察："Agent 的很多魔法来自于它能够从错误信息中修正自己并迭代"
  - 有效的反馈工具比更好的初始提示更能提升成功率
- ### 工作流集成策略

  **即时任务委派：**
  - 通过 Slack 等渠道直接路由常规请求（bug 修复、小功能）给 Agent
  - 利用 Agent 做探索性工作的同时保持对主要职责的专注
  - 移动端访问支持出行时的紧急修复

  **消除日常琐事：**
  - 委派重复性任务：commit bisect、文档更新
  - 同时为多个竞争架构方案创建原型，避免"分析瘫痪"（analysis paralysis）
- ### 中大型任务管理（1-6 小时任务）

  文章特别强调这个时间段具有**最高 ROI**。核心策略：

  #### 检查点架构（Checkpoint Architecture）
  结构化工作流：
  ```
  计划 → 实现分块 → 测试 → 修复 → 检查点审查 → 下一分块
  ```
  - 确保在问题跨越数据库、后端、前端多层级级联之前对齐
  - 每个检查点由人类审查，确认方向正确后再继续

  #### 协作规划（Collaborative Planning）
  - 对于模糊任务，先让 Agent 做发现性提问："我们的认证系统是如何工作的？"
  - 部分 Agent 平台提供专门的"规划模式"，强调先读现有代码而非立即修改
  - 先理解再动手，减少后期返工

  #### 现实预期
  - 目标是"约 80% 的时间节省，而非完全自动化"
  - 最终质量保证仍然不可或缺
- ### 高级自动化方案

  **可复用提示模板（Playbooks）：**
  为以下场景创建模板：
  - Feature flag 清理
  - 依赖升级
  - 新 PR 的测试添加/修复

  **智能代码审查：**
  - 让 Agent 对照团队常见错误清单检查 PR
  - 特别适合传统 linting 规则无法捕捉的模式

  **事件响应集成（Incident Response）：**
  - 通过 API 或 CLI 在生产告警时触发 Agent
  - 但要注意局限性："AI 的调试能力并不强"
  - 正确用法：让 Agent"标记最可疑的错误和变更"，而非要求端到端修复
- ### 环境定制（Environmental Customization）

  **环境对齐：**
  让 Agent 环境精确匹配团队配置：
  - 语言版本和包依赖
  - Pre-commit hooks
  - 通过 `.envrc` 或 `.bashrc` 预认证浏览器登录

  **自定义工具：**
  - 除 MCP 外，为常见工作流创建简单的 CLI 脚本
  - 示例：一个只暴露第一个失败测试和详细错误信息的脚本，可显著提升 Agent 的专注度和完成率

  **知识库集成：**
  - 在 Agent 持久化知识库中记录团队常见错误和项目架构
  - 文档化具体流程（如添加服务路由）所需的所有 boilerplate 位置
- ### 关键局限性（Critical Limitations）

  **调试约束：**
  - Bug 通常需要数据库/日志访问和复杂推理
  - 应让 Agent 提供"可能根因列表"，而非要求完整诊断

  **视觉推理缺口：**
  - 当前模型在匹配设计稿或截图方面表现欠佳
  - 解决方案：使用可复用组件的代码化设计系统

  **知识截止日期问题：**
  - Agent 会因为训练数据截止日期而使用旧版 API 模式
  - 解决方案：每次都显式引用最新文档
- ### 何时放弃与重启（When to Abandon and Recover）

  **早期终止信号：**
  - Agent "忽视指令" → 立即停止
  - Agent "在圈子里打转" → 立即停止
  - 这些信号表明任务复杂度超出 Agent 当前能力，不是可纠正的错误

  **重新开始优于迭代：**
  - 当反馈循环停滞时，用整合后的指令重新开始"往往能更快到达成功"
  - Agent 擅长生成全新代码，但在修复已损坏的环境时表现差

  **多样化实验：**
  - 初期尝试不同提示方式
  - 在自然成功的任务类型上加大投入，在表现不佳的类型上"及时止损"
- ### 安全与权限（Security & Permissions）

  - 为安全测试创建一次性邮箱账号
  - 只使用开发/预发布环境
  - 为云资源访问分配自定义 IAM 角色
  - 尽可能使用只读 API key
  - 影响外部服务的脚本必须由人类手动执行
- ### 人类角色的演变

  技术专业知识依然不可替代。随着 Agent 处理日常工作，工程师逐渐转向：
  - 监督多个并行系统
  - 维持深度理解和判断力
  - **所有权和验证变得更重要，而非更次要**
  - 自动化放大了个人影响力，对判断力的要求也随之提高
