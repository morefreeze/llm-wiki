# 每次会话结束前都做好交接（Learn Harness Engineering L12）

> **一句话**：熵增是默认状态（Lehman 定律）——清洁状态是会话完成的必要条件，不是可选善后工作；"以后再清理"等于永远不清理。

## 关键观点

### 1. 熵增是默认状态

Lehman 软件演化定律：持续变更的系统，除非主动管理，否则复杂性必然增加。OpenAI Codex 实验观察：agent 会复制仓库中已有模式，即使那些模式是次优的。没有清洁策略的项目 12 周数据：

| 时间点 | 构建通过率 | 测试通过率 | 新会话启动时间 |
|--------|-----------|-----------|--------------|
| 第 1 周 | 100% | 100% | 5 分钟 |
| 第 4 周 | 95% | 92% | 15 分钟 |
| 第 8 周 | 82% | 78% | 35 分钟 |
| 第 12 周 | 68% | 61% | 60+ 分钟 |

有清洁策略的同一项目：第 12 周构建 97%、测试 95%、启动 9 分钟。

### 2. 清洁状态五个维度

| 维度 | 说明 |
|------|------|
| 构建通过 | 代码能无错构建——下一个会话不应先修构建错误 |
| 测试通过 | 所有测试（含会话前已有测试）通过，CI 环境验证 |
| 进度已记录 | 机器可读工件记录完成/进行中/待开始子任务（减少 60-80% 启动诊断时间） |
| 无过时工件 | 调试日志、临时文件、注释掉的代码、TODO 标记已清理 |
| 启动路径可用 | 标准启动路径可用，下一个会话能不人工干预就开始工作 |

### 3. 会话完整性（数据库事务类比）

类比数据库事务：要么全部提交并留下清洁状态，要么回滚到上一致状态，没有中间地带。

### 4. 双模式清理策略

**即时清理（每个会话结束时）**：清理本次会话临时工件、更新功能清单状态、确保构建和测试通过。引用计数式——用完就清。

**定期清理（每周一次）**：全系统扫描、处理累积结构性问题、更新质量文档、运行基准测试检测漂移。追踪式——定期大扫除。

### 5. Harness 简化（随模型能力提升）

Anthropic 实验：最初 harness 包含 sprint 拆分机制（把工作分成小块让 Sonnet 4.5 逐个完成）。Opus 4.6 发布后，模型原生能力可自主处理工作分解，sprint 构造变成不必要开销，移除后 builder agent 能连续工作 2 小时以上不跑偏。

**推荐做法**：每月挑一个 harness 组件暂时禁用，跑基准任务。无退化则永久移除，有退化则恢复。

### 6. 高吞吐量 Merge 哲学

在 agent 每天产出远超人类审查能力的环境里（如 3.5 个 PR/天），最小化阻塞式 merge gate 是正确的。修正成本低于等待审查成本时，快速前进 + 快速修正优于缓慢确认。
