- # 清洁状态（Clean Session State）
  
  熵增是默认状态（Lehman 软件演化定律）——清洁状态是会话完成的必要条件，不是可选善后工作，是"完成定义"的一部分。"以后再清理"等于永远不清理。
- ## 清洁状态五个维度（缺一不可）
  
  | 维度 | 说明 |
  |------|------|
  | **构建通过** | 代码能无错构建——下一个会话不应先修构建错误 |
  | **测试通过** | 所有测试（含会话前已有测试）通过，CI 环境验证而非"在我机器上通过" |
  | **进度已记录** | 机器可读工件记录完成/进行中/待开始子任务（减少 60-80% 启动诊断时间） |
  | **无过时工件** | 调试日志、临时文件、注释掉的代码、TODO 标记已清理 |
  | **启动路径可用** | 标准启动路径可用，下一个会话能不人工干预就开始工作 |
- ## 会话完整性
  
  类比数据库事务：要么全部提交并留下清洁状态，要么回滚到上一致状态，**没有中间地带**。
- ## 熵增的量化数据
  
  没有清洁策略的项目 12 周数据：
  
  | 时间点 | 构建通过率 | 测试通过率 | 新会话启动时间 |
  |--------|-----------|-----------|--------------|
  | 第 1 周 | 100% | 100% | 5 分钟 |
  | 第 4 周 | 95% | 92% | 15 分钟 |
  | 第 8 周 | 82% | 78% | 35 分钟 |
  | 第 12 周 | 68% | 61% | 60+ 分钟 |
  
  有清洁策略的同一项目第 12 周：97%、95%、9 分钟。12 周后构建通过率差 29 百分点，启动时间差 85%。
- ## 双模式清理策略
  
  **即时清理（每个会话结束时）**：清理本次会话临时工件、更新功能清单状态、确保构建和测试通过。引用计数式——用完就清。
  
  **定期清理（每周一次）**：全系统扫描、处理累积结构性问题、更新质量文档、运行基准测试检测漂移。追踪式——定期大扫除。
- ## 质量文档
  
  对每个模块持续评分的活跃工件（不是一次性评估），让新会话知道优先处理哪里：
  
  ```markdown
  ## 支付模块 (质量: C)
  - 验证通过: 部分（支付回调未测试）
  - agent 可理解: 困难（逻辑分散在 3 个文件）
  - 测试稳定性: 不稳定（2 个 flaky 测试）
  ```
- ## Harness 简化原则
  
  随着模型能力提升，定期移除不再必要的 harness 组件。Anthropic 实验：Opus 4.6 发布后，sprint 拆分机制变成不必要开销，移除后 builder agent 连续工作 2 小时以上不跑偏，反而更流畅。
  
  **推荐做法**：每月挑一个 harness 组件暂时禁用，跑基准任务。无退化则永久移除。
  
  > "随着模型改进，harness 的有趣组合不是变少了，而是移动了。"
- ## 幂等清理
  
  清理操作无论执行多少次都产生相同结果，确保在失败重试场景中仍然安全：
  ```bash
  rm -f /tmp/debug-*.log   # -f 确保文件不存在时不报错
  git checkout -- .env.local  # 恢复到已知状态
  npm run test             # 验证清理未破坏功能
  ```
