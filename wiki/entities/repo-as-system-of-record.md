# 仓库即规范（Repo as System of Record）

Agent 需要的所有知识必须在仓库里可发现，而不是散落在脑子里、Slack 消息里、或者某人的本地环境里。仓库即规范——这是 harness 工程的基础假设。

## 知识可见性缺口

**知识可见性缺口 = 团队实际拥有的知识 − agent 通过查看仓库能访问到的知识**

缺口越大：
- agent 需要越多人类介入
- agent 犯同样错误的频率越高
- 发现成本（Discovery Cost）越高

## 冷启动测试：判断仓库质量的 5 问

给全新 agent 会话看仓库内容（不给任何口头上下文），让它回答：

1. 这个项目是干什么的？（一句话）
2. 怎么把项目跑起来？（具体命令）
3. 怎么跑测试并确认都通过了？
4. 这周正在进行什么工作？
5. 下一步该做什么？

答不上来的问题，其答案就在知识缺口里。

## 知识衰减率

没有明确维护策略的文档，以约 **20% 的月衰减率**失去准确性——每过一个月，变成错误信息的概率增加 20%。

## 四个让仓库成为事实来源的原则

| 原则 | 实践 |
|------|------|
| **知识靠近代码** | 文档和描述的代码放在一起，`src/auth/` 旁边放 `src/auth/CONTEXT.md` |
| **标准化入口** | 有且只有一个"从这里开始"的入口（AGENTS.md 或 README） |
| **最小但完备** | 只放 agent 完成任务所必需的知识，质量优于数量 |
| **和代码一起更新** | 每次改代码，同时更新相关文档，纳入 PR 完成标准 |

## 用 ACID 原则管理 Agent 状态

Agent 的状态文件（PROGRESS.md、DECISIONS.md）应像数据库事务一样管理：

- **原子性**：状态更新是完整的，不存在"更新了一半"的状态
- **一致性**：状态文件始终反映代码库的真实状态
- **隔离性**：不同 agent 会话不互相干扰状态
- **持久性**：状态变更在 git 提交后永久保存

## 实战数据

30 个微服务的电商平台，改造后（所有微服务有标准化的 AGENTS.md）：
- 需要人工介入的任务比例降低 **60%**
- 单个 agent 会话内完成的任务比例提升 **40%**
