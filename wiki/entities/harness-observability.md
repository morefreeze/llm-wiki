# Harness 可观测性（Harness Observability）

没有可观测性，agent 在不确定状态中做决策——评估变成主观判断，重试变成盲目摸索。可观测性缺失导致 **30-50%** 的会话时间浪费在重复诊断上。

## 双层可观测性

**运行时可观测性**：系统层信号（日志、追踪、进程事件、健康检查）。回答"系统做了什么"。类比：仪表盘（速度、油量、发动机温度）。

**过程可观测性**：harness 决策工件可见性（计划、评分标准、验收条件）。回答"为什么这个变更应该被接受"。类比：导航系统（知道在哪、为何走这条路）。

两层缺一不可，相互增强：运行时信号解释行为，过程工件解释意图。

## 可观测性缺失的四类问题

| 问题 | 说明 |
|------|------|
| 无法区分"正确"和"看似正确" | 只有运行时追踪能揭示执行路径偏离预期 |
| 评估变成玄学 | 缺评分标准，同一输出不同评估者截然不同评价 |
| 重试变成盲猜 | 不知为何失败，随机重试方向消耗 token 和时间 |
| 会话交接信息断崖 | 新会话从零诊断系统状态，占总时间 30-50% |

## 核心概念

**任务轨迹（Task Trajectory）**：任务从开始到完成的完整决策路径记录，类似分布式系统请求追踪。每一步操作及其上下文都被记录——就像黑匣子，出了问题可以回放完整过程。

**冲刺合同（Sprint Contract）**：编码开始前由生成者和评估者协商的短期协议：
```markdown
# 冲刺合同: 暗色模式支持
## 范围
- 修改主题切换组件、更新全局 CSS 变量、添加暗色模式测试
## 验证标准
- 每个组件的视觉回归测试通过、主流程 E2E 测试通过、无样式闪烁
## 排除项
- 不处理打印样式、不处理第三方组件暗色模式
```

防止"生成者做了评估者因可预见原因立即拒绝的东西"——施工协议要在开工前签。

**评估评分标准**：把质量评估从主观判断变成基于证据的结构化评分，使不同评估者对同一输出产生相似结论。

## Anthropic 三 Agent 实验

"用 Web Audio API 做一个浏览器端 DAW"，三个 agent 各司其职：

| Agent | 职责 | 可观测性角色 |
|-------|------|------------|
| Planner | 把 1-4 句需求扩展成完整产品规格 | 生成过程工件 |
| Generator | 按 sprint 逐功能实现 + 自评 | 产生运行时信号 |
| Evaluator | Playwright MCP 点击测试 + 四维度评分 | 消费两种信号，输出结构化评分 |

总计：3 小时 50 分钟，$124.70。核心功能可用（对比单 agent 裸跑：20 分钟，$9，不可用）。

## OpenTelemetry 标准化

每个 harness 会话创建一个 trace，每个任务创建一个 span，每个验证步骤创建子 span，使用标准属性标注关键信息，可接入 Jaeger/Zipkin 等标准工具链。
